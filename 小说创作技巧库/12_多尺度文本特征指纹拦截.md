# 12_多尺度文本特征指纹拦截

> 提炼自 ICLR 2024 "Multiscale Positive-Unlabeled Detection of AI-Generated Texts" 项目的三个命令行参数（multiscale_kit / en_cleaning / pu_type）所对应的宏观统计思想，翻译为纯正则表达式和 Python 可执行卡口。不涉及原论文源码。

## 一、多尺度词链嗅探（平替 multiscale_kit）

检测器在多个文本粒度上计算词序列的条件概率。逆翻译到生成侧：AI 生成文本的词组级共现概率极度过剩——模型总是把高概率词对（"不仅...更是""拉开了...序幕""见证了...交织"）当作安全填空塞进段落。这些词组单独看每一个字都合法，但全文中词组的共现频率远高于人类写作——人类作者不会在三段内重复使用同一对关联词。

### 词链黑名单（双字至六字级）

以下词组在每一千字中累计出现超过一次即触发警告级违规：

双字级：赋能 抓手 闭环 对齐 拉通 打通 链路 颗粒度 维度

三字级：不仅仅是 更是要 可以说 可谓是 不难看 不难发 由此可 通过以 综上所

四字级：不仅仅是更 拉开了序幕 见证了历史 在这片土地 不可否认的 显而易见的 值得注意的 深入探讨了

五字级：不仅仅是更是 拉开了新序幕 见证了这一历 在这片土地上 不可否认的是 显而易见的是

六字级及以上：不仅仅是更是对 拉开了全新序幕 见证了这一历史 在这片古老土地上

### 词链嗅探的正则扫描逻辑

Python 审计模块在收到 Kimi 初稿后执行以下扫描，不依赖任何外部模型：

1. 对每个禁词链在全文计次。单条词链重复出现超过一次——警告。单条词链重复出现超过三次——升级为致命。
2. 相邻二十字范围内出现超过两个不同的禁词链——警告。表明这一段是 AI 在"词组堆叠模式"下生成的。
3. 全文禁词链总出现次数除以千字字数——每千字超过三处，审计报告标注"词链密度超标"。

### 本小说实际应用场景

沈节在军机书房听到算盘声传来。写这段时不得出现"算盘声如同某种命运的节拍"——"命运的节拍"是 AI 高概率词组。正确写法："隔壁账房的算盘还在响——木珠在竹签上来回撞。每次撞到底档的时候木珠和木框之间有一声很短的闷响。沈节听着这声闷响重复了大概四十多次——然后停了。他等了一息。算盘声没有再响起来。隔壁的人大概对完账了。"这一段全是物理声音的逐次记录，没有任何词组落在禁词链黑名单上。

李观照在附子毒发时赶到军机书房——写这段时不得出现"她的眼中交织着心痛与无力"。"交织着"是 AI 高概率词链。正确写法："她推开门的时候沈节正趴在桌面上——不是晕了，是痛得整个人蜷在桌面上，左手压在左胸口，右手还在账册第十四页的日期旁边。那根碳条还没从手指间掉。她把碳条抽出来。沈节的手指在她抽碳条的时候攥了一下——不是不放，是痛到手指不听使唤了。她把碳条放到桌上。沈节的手指慢慢松开了。"

## 二、格式与分栏指纹洗白（平替 en_cleaning）

en_cleaning 在原论文中用于清理 HC3 数据集中人类文本的冗余空格——AI 不会在标点前后留不规则空格，这种"过于干净"本身就是检测信号。逆翻译到生成侧：AI 除了标点干净之外，还有一套习惯性的"文本排版指纹"——前置客套话、后置总结尾音、括号内的剧本式动作批注、居中式分点陈列——这些在人类自然写作中很少同时出现。

### 格式指纹黑名单

以下排版特征在 Kimi 初稿中被正则匹配到即触发警告：

1. 前置客套话：首段以"好的，""明白了，""让我来写""以下是根据""为您创作"开头——直接删除首段。正文从第二段开始。
2. 后置总结尾音：末段以"这意味""由此可见""综上""总而言之""通过以上""到这里""说明了""启示""告诉我们"开头——直接删除末段或末句。
3. 括号内剧本式批注：正文中出现小括号内含动作描写——如"（他苦笑）（她眼神一黯）（沈节心中一阵绞痛）"。正则匹配到中英文括号内连续三字以上——标记为格式指纹，删除括号及括号内全部内容，如括号内动作有实际信息需改写为物理描述句。
4. 居中式分点："首先，……其次，……最后，……"——正则匹配到连续三组分号结构——警告。
5. 前置称号与自我署名："作为一部""作为一本""我叫""我是"——不出现在叙事中。

### 格式指纹清洗的正则扫描逻辑

Python 审计模块依次执行：
```
re.match(r'^(好的|明白了|让我来写|以下是根据|为您创作)', text) -> strip first paragraph
re.search(r'[（(][^）)]{3,}[）)]', text) -> strip parenthetical stage directions
re.search(r'(这意味|由此可见|综上|总而言之|通过以上|到这里|说明了|启示|告诉我们)', last_paragraph) -> strip last paragraph
re.findall(r'(首先.*其次.*最后)', text) -> flag three-part enumeration
```

每条命中生成一条警告。累计三条触发驳回。注意：括号内批注是最常见的 Kimi 指纹——Kim 在生成中文小说文本时习惯在角色动作后用小括号添加情绪批注，这是训练数据中大量剧本/网文文本的统计特征泄漏。

## 三、动态审计权重（平替 pu_type 逻辑）

pu_type = dual_softmax_dyn_dtrun 在原论文中的含义是：正类（AI 文本）和非标注类（可能人类可能 AI）之间使用双重 softmax 做动态阈值调整——即检测器不是用一个固定的阈值来判定"AI 还是人类"，而是根据当前样本的特征分布实时调整。逆翻译到审计侧：如果前一轮的正文触发了驳回，说明 Kimi 在这一场景的文本生成中陷入了某种默认高概率模式——此时需要对下一轮的审计标准执行动态收紧。

### 动态阈值调整表

| 驳回轮次 | 句长变异系数门槛 | 词链密度上限（每千字） | 数目字下限 |
|----------|-----------------|---------------------|-----------|
| 首轮（无驳回历史） | 0.60 | 3 处 | 2 个 |
| 第一轮驳回后 | 0.70 | 2 处 | 3 个 |
| 第二轮驳回后 | 0.80 | 1 处 | 4 个 |
| 第三轮（强制通过后备） | 0.80 | 1 处 | 4 个 |

每次 Kimi 重新提交正文——Python 审计模块从 AuditResult 中读取上一轮的驳回次数，按上表调整 CV_THRESHOLD/FATAL_WORDS_LIMIT/NUMERIC_MIN。第三轮因已达上限，不再驳回——仅记录全部违规项入库供人工复查。
